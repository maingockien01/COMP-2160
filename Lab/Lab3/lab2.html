Before you start coding on this problem, consider for yourself:

  1. What is the input or inputs to this problem? How is it formatted?
  2. What does the output look like?
  3. How might the input or inputs be transformed into the output?

Next, look carefully at the skeleton code that’s provided. Yes, it’s C, it’s different. Trace the code mentally. Use a piece of paper and a pencil. What does the current code do with or to the inputs?

Exercise

Do problem 6 of Column 3. Here are test files for the template and data, as given on page 24. Here’s some skeleton code to help get you started – the point is for you to do the input processing. Also note that the textbook provides a pseudo-code solution for this problem.

The easiest way to process the customer data is to tokenize it. You’ll have to store the data in an array – the skeleton defines this for you, and assumes that there are always 9 fields and a maximum of 30 characters per field (don’t worry about error checking for the lab…).

The template file is best processed character by character looking for variable markers (the $ sign). Note that this is the exact same processing that printf() uses when processing the formatting string. You have to process the file for each customer – it’s easier to re-read the file repeatedly rather than store it in a buffer (the system has it buffered anyway). Note how the skeleton code doesn’t open/close the file, it uses the rewind() function (see man rewind).
